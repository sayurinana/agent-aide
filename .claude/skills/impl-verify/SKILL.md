# impl-verify 指南

## 目标

把 `todo.md` 中的实施任务点逐个做完，并在每个任务点完成后立刻做最小充分验证。

这里的重点不是“先堆一批改动再统一检查”，而是保持小步推进、即时审验、及时回写。

## 进入前先读

实施前至少读取：

- `information.md`
- `design.md`
- `todo.md`
- `task-summary.md`

必要时补充读取相关代码、测试、接口定义和 memory 文档。

## 核心方法

### 1. 只拿下一个任务点

1. 在 `todo.md` 中定位当前未完成的任务点。
2. 优先选择一个可独立闭环的最小任务点。
3. 只有在多个子项强耦合、拆开会增加风险时，才允许成组处理。

不要一次跨越多个未完成任务点后再回头补勾选。

### 2. 实施

围绕当前任务点完成代码、配置、测试、脚本或文档改动。

实施时始终回看：

- 是否仍符合 `information.md` 的目标和边界
- 是否仍符合 `design.md` 的方案和约束
- 是否引入了新的隐含前提、迁移风险或用户未确认的行为变化

### 3. 立即审验

每完成一个任务点后，立刻执行与该任务点匹配的最小充分验证，例如：

- 单元测试
- 集成测试
- 构建、编译、lint
- CLI 或接口手工验证
- 对关键输出做静态检查

验证要和本次变更直接对应，不能用“项目大体能跑”代替当前任务点的完成证明。

### 4. 审验结果处理

- 审验通过：
  - 勾选 `todo.md` 中对应任务点
  - 更新 `task-summary.md`，保持摘要简洁且反映最新进度
  - 再进入下一个任务点
- 审验未通过：
  - 留在当前阶段内修复
  - 重新验证，直到通过
- 发现问题不是实现细节，而是需求、设计或阶段选择本身有偏差：
  - 停止继续堆改动
  - 进入 `rework` skill 处理返工

## 循环模式

当 `todo.md` 中阶段声明包含 `impl-verify:loop` 时，表示该阶段需要重复执行以下循环：

1. 选取下一个未完成任务点
2. 实施
3. 立即审验
4. 回写 `todo.md` 与 `task-summary.md`
5. 继续下一个任务点

只有当本阶段应完成的实施任务点全部完成后，才进入后续阶段。

如果阶段声明是 `impl-verify` 而不是 `impl-verify:loop`，也仍然需要即时审验，只是流程上不强调循环语义。

## 回写要求

### todo.md

- 真实勾选已通过审验的任务点
- 不要提前勾选
- 不要把“已写代码但未验证”当作完成

### task-summary.md

- 继续保持一级标题为简短任务摘要
- 正文只补充关键进展、剩余风险和下一步提醒
- 尽量维持 10 行以内、每行 30 字以内

## 返工触发信号

出现以下情况时，不应继续硬做，应考虑进入 `rework`：

- 用户反馈显示当前方向做错了
- `design.md` 的方案已被事实证明不可行
- 需要新增或删除阶段
- 新增约束会影响已完成任务点
- 当前任务点无法在原边界内闭环

## 完成条件

`impl-verify` 阶段完成时，应满足：

- 当前阶段负责的任务点都已经过验证并勾选
- `task-summary.md` 已反映最新状态
- 没有隐藏的未验证改动
- 已明确是进入下一个阶段，还是因为返工回退
