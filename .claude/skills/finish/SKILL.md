# finish 指南

## 目标

把一个已经经过用户确认的正式任务做完整收尾，而不是只结束当前对话。

`finish` 处理的是任务生命周期结束；
`/aide:bye` 处理的是本轮会话离场。

## 进入条件

进入 `finish` 前应满足：

- 用户已经确认当前结果可接受
- `todo.md` 中应完成的任务点都已真实勾选
- `task-summary.md` 已更新为最终状态
- 需要同步的文档、memory 或说明已补齐，或已明确记录为什么暂不处理

如果这些前提不成立，先回前面阶段，不要硬收尾。

## 收尾流程

### 1. 先做最终检查

确认以下事项与真实状态一致：

- 任务结果与用户确认内容一致
- 任务文档没有遗留的“待补充”“稍后再说”式占位
- 本次验证结果已保留下来，能解释为什么可以收尾
- 若存在已知限制，已对用户透明说明

### 2. 用 aide 程序完成任务生命周期收束

优先使用 aide 程序而不是手写一套临时流程。

至少覆盖两件事：

- 完成任务分支向常驻分支的收束或合并
- 完成任务目录归档

实践上以当前项目已实现的 aide 命令为准：

- 如果项目当前使用阶段追踪命令推进到 `finish`，先完成该动作，并确认程序输出的合并结果
- 在常驻分支上执行 `aide archive <编号>`；若当前只有一个未归档任务，可按程序约定直接执行 `aide archive`

不要跳过 aide 的状态更新，直接手动移动目录或私自修改状态文件。

### 3. 同步全局信息

根据任务影响范围，更新必要的全局信息，例如：

- `aide-memory/memory/overview.md`
- 相关概念、结构或流程说明
- 其他会影响后续任务接续判断的长期信息

原则是只同步“后续任务还需要继续知道”的信息，不把临时过程噪音写进 memory。

### 4. 判断是否还要执行 `/aide:bye`

- 如果任务生命周期已经结束，但本轮对话还要继续其他工作，可以不立即离场
- 如果本轮工作也结束了，再走 `/aide:bye` 做会话级收尾

## 风险控制

出现以下情况时，不应继续 finish：

- 用户确认还不明确
- 当前仍在任务分支，但项目要求先通过 aide 的阶段收束动作
- `aide archive` 前置条件未满足
- memory 需要同步的重要变化还没落盘

## 完成条件

`finish` 阶段完成时，应满足：

- 任务已经通过 aide 程序完成收束
- 任务目录已进入归档区
- 需要保留的长期信息已同步
- 用户知道任务已正式结束，以及下次应从哪里接续
