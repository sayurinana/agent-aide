# review 指南

## 目标

对已完成的工作进行系统性审查，包括代码质量评估、文档审校、方案评审，确保交付物符合最佳实践和质量标准。

## 进入前先读

- `information.md` — 任务目标与边界
- `design.md` — 方案细节
- `todo.md` — 已完成的任务点
- `task-summary.md` — 实施摘要
- 相关代码文件和文档

## 审查类型

review 可涵盖三种审查，根据任务特性选择：

### 代码审查

评估代码质量：

| 维度 | 检查项 |
|------|--------|
| 可读性 | 命名清晰、结构分明、注释恰当 |
| 可维护性 | 模块化、解耦、单一职责 |
| 性能 | 无明显瓶颈、资源使用合理 |
| 安全性 | 无常见漏洞、敏感数据处理正确 |
| 最佳实践 | 符合项目规范、遵循设计模式 |

### 文档审校

审校文档准确性：

| 维度 | 检查项 |
|------|--------|
| 准确性 | 内容与实现一致 |
| 完整性 | 覆盖必要信息 |
| 一致性 | 格式、术语、风格统一 |
| 可读性 | 结构清晰、表达简洁 |

### 方案评审

评审设计可行性：

| 维度 | 检查项 |
|------|--------|
| 可行性 | 技术方案可实现 |
| 合理性 | 方案符合需求、权衡得当 |
| 风险 | 潜在风险已识别、有缓解措施 |

## 审查流程

### 1. 确定审查范围

根据任务类型选择审查重点：

- 代码任务 → 代码审查为主
- 文档任务 → 文档审校为主
- 设计任务 → 方案评审为主
- 综合任务 → 多类型审查

### 2. 执行审查

逐项检查并记录结果：

```markdown
## 审查记录

### 代码审查
- [x] 命名规范符合项目约定
- [x] 模块职责单一
- [ ] 性能：存在 N+1 查询问题（建议优化）

### 文档审校
- [x] README 与实际功能一致
- [x] API 文档参数说明完整
```

### 3. 问题分级

审查发现的问题按严重程度分级：

| 级别 | 说明 | 处理方式 |
|------|------|----------|
| 阻塞 | 影响功能、安全、稳定性 | 必须修复后才能进入 confirm |
| 建议 | 可优化但不阻塞 | 记录待办，可延后处理 |
| 信息 | 改进建议、最佳实践提醒 | 记录备忘，不强制 |

### 4. 处理阻塞问题

阻塞问题必须处理：

1. 记录问题详情
2. 在当前阶段修复
3. 重新审查确认解决
4. 继续后续审查

### 5. 完成确认

审查完成后：

- 更新 task-summary.md 记录审查结论
- 阻塞问题已解决
- 建议/信息项已记录在 todo.md 待办
- 进入 docs-update 或 confirm 阶段

## 与其他阶段的关系

```
impl-verify → integration → review → docs-update → confirm
                    ↑                ↓
                  review → confirm（无 docs-update 时）
```

review 是可选阶段，仅在 build-task 阶段选定时执行。

## 代码审查详细指南

### 可读性检查

- 变量/函数命名是否表达意图
- 代码结构是否层次分明
- 关键逻辑是否有注释
- 复杂表达式是否可拆解

### 可维护性检查

- 模块边界是否清晰
- 依赖关系是否合理
- 是否存在重复代码
- 是否符合单一职责

### 性能检查

- 是否存在不必要的循环
- 数据结构选择是否合理
- 是否存在资源泄漏
- 缓存策略是否正确

### 安全性检查

- 输入验证是否完整
- 权限检查是否到位
- 敏感数据是否加密
- 错误处理是否安全

## 文档审校详细指南

### 准确性检查

- 描述与实际实现是否一致
- 参数/配置说明是否正确
- 示例是否可运行
- 版本信息是否更新

### 完整性检查

- 必要章节是否齐全
- 关键信息是否遗漏
- 使用前提是否说明
- 错误处理是否覆盖

## 完成条件

review 阶段完成时，应满足：

- 审查范围已确定并执行
- 阻塞问题已解决
- task-summary.md 已更新审查结论
- 建议/信息项已记录
- 明确进入下一阶段（docs-update 或 confirm）
