## MODIFIED Requirements

### Requirement: 正式收尾与归档闭环

系统 SHALL 在用户确认通过后，支持任务从完成状态进入正式收尾、归档、分支收束与 memory 同步闭环，并明确区分“暂停离场”与“正式结束”。

#### Scenario: finish 串联正式收尾闭环
- **WHEN** 任务已通过成果确认并进入 `finish` 阶段
- **THEN** 系统指导或执行正式收尾动作
- **AND** 完成任务归档、分支收束与需要长期保留的信息同步

#### Scenario: archive 更新目录、分支映射与生命周期状态
- **WHEN** 用户执行正式归档动作
- **THEN** 系统同步更新任务目录位置、分支映射状态与任务生命周期状态
- **AND** 保持这些产物之间的一致性

#### Scenario: `aide bye` 不替代正式 finish
- **WHEN** 用户在任务分支执行 `aide bye`
- **AND** 当前任务已经满足 finish 条件
- **THEN** 系统将其视为需要进入正式收尾路径的任务
- **AND** 不把一次普通离场直接记为任务结束

### Requirement: 草案冲突与任务冲突保护

系统 SHALL 在草案残留、任务并存或状态冲突时给出明确保护和提示，避免用户误将冲突状态继续推进。

#### Scenario: 草案残留时阻止误判为干净状态
- **WHEN** `task-now.md` 存在未处理变更或 `task-now/` 目录残留
- **THEN** 系统明确提示当前存在草案冲突或未完成草案
- **AND** 不将项目判定为可直接开始新任务的干净状态

#### Scenario: 多任务并存且存在草案残留时给出明确接续提示
- **WHEN** 项目存在多个未归档任务
- **AND** 同时存在草案残留
- **THEN** 系统向用户展示可接续任务与草案冲突信息
- **AND** 明确哪些动作会影响当前草案或当前任务状态
