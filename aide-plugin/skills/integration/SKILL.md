# integration 指南

## 目标

在所有任务点完成 impl-verify 循环后，执行整体集成测试，确保各改动部分能够协同工作，并检查可能遗漏的边界情况和兼容性问题。

## 进入前先读

- `information.md` — 任务目标与边界
- `design.md` — 方案细节
- `todo.md` — 已完成的任务点
- `task-summary.md` — 实施摘要

## 与 impl-verify 的区别

| 维度 | impl-verify | integration |
|------|-------------|-------------|
| 范围 | 单个任务点 | 全部改动集成 |
| 目标 | 验证当前改动正确 | 验证整体协同工作 |
| 方式 | 即时验证 | 系统性测试 |

impl-verify 保证"每个零件是对的"，integration 保证"零件组装后整体是对的"。

## 测试类型

### 1. 功能集成测试

验证各功能模块协同工作：

- 功能链路完整性
- 模块间接口正确
- 数据流转无丢失

### 2. 边界情况测试

检查可能遗漏的边界：

- 空值、极端值处理
- 异常路径覆盖
- 并发、竞态场景

### 3. 兼容性测试

检查与现有系统的兼容：

- 向后兼容性
- 接口变更影响
- 配置迁移影响

### 4. 环境测试

在真实或模拟环境中验证：

- 部署流程正确
- 配置生效正确
- 外部依赖正常

## 测试流程

### 1. 制定测试计划

根据 design.md 和 todo.md 列出：

- 需要验证的集成点
- 需要覆盖的边界情况
- 需要检查的兼容性项

### 2. 执行测试

按计划逐一测试，记录结果：

```markdown
## 集成测试记录

### 功能集成
- [x] 用户登录 → 权限检查 → 数据访问链路正常
- [x] API 接口与数据库操作协同正常

### 边界情况
- [x] 空用户名处理正确
- [ ] 并发登录场景待补充（非阻塞问题）

### 兼容性
- [x] 旧版本配置文件可正常加载
```

### 3. 问题处理

- 测试通过：勾选记录，继续下一项
- 测试失败：记录问题，评估严重程度
  - 阻塞问题：立即修复后重测
  - 非阻塞问题：记录在 todo.md 待办，继续测试

### 4. 完成确认

全部测试完成后：

- 更新 task-summary.md 记录集成测试结论
- 非阻塞问题在 todo.md 中标记待办
- 进入 review 或 confirm 阶段

## 与其他阶段的关系

```
impl-verify → integration → review → confirm
                    ↓
                  confirm（无 review 时）
```

integration 是可选阶段，仅在 build-task 阶段选定时执行。

## 发现阻塞问题时的处理

当集成测试发现阻塞问题：

1. 记录问题详情和影响范围
2. 判断问题类型：
   - 实现细节问题：修复后重新测试
   - 设计缺陷：考虑是否需要 rework
   - 需求偏差：进入 rework 评估返工
3. 修复后重新执行集成测试

## 完成条件

integration 阶段完成时，应满足：

- 集成测试计划已执行
- 阻塞问题已解决
- task-summary.md 已更新
- 非阻塞问题已记录在 todo.md 待办
- 明确进入下一阶段（review 或 confirm）
