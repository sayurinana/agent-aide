# confirm 指南

## 先澄清名称

这里的 `confirm` 指正式任务执行后的成果确认阶段。

它不是 `aide confirm` 命令。

- `aide confirm`：把 `task-now/` 草案敲定为正式任务
- `confirm` skill：在任务接近完成时向用户展示成果并收集确认意见

不要混淆这两者。

## 目标

让用户基于真实结果、真实验证证据和真实限制做确认，而不是只听口头总结。

## 进入前准备

进入本阶段前，应先整理好：

- 已完成的 `todo.md`
- 最新的 `task-summary.md`
- 本次改动摘要
- 已执行的验证项及结果
- 仍存在的限制、风险或待办

如果这些信息还不完整，先补齐，再进入确认。

## 展示方式

向用户汇报时，至少覆盖以下内容：

1. 做了什么
2. 改动影响了哪里
3. 如何验证已经生效
4. 还有哪些边界、限制或未覆盖项

汇报要基于事实，不要把猜测包装成结论。

## 反馈收集方法

确认阶段至少要把以下问题问清：

- 结果是否符合预期
- 是否还有遗漏场景或边界条件
- 是否接受当前实现进入收尾

如果任务较复杂，可以把反馈拆成以下几个维度收集：

- 功能是否符合预期
- 交互或输出是否符合预期
- 质量风险是否可接受
- 是否需要返工或补充阶段

## 反馈分流

### 用户确认通过

当用户明确表示结果可接受时：

- 记录确认结论
- 保持 `todo.md`、`task-summary.md` 与实际结果一致
- 进入 `finish`

### 用户提出小修正

如果只是局部修正、且不需要改变需求理解或阶段流程：

- 在当前上下文中补做修正
- 重新给出更新后的结果摘要
- 再次确认

### 用户提出返工级问题

如果反馈涉及以下情况，应转入 `rework`：

- 需求理解偏差
- 设计方向错误
- 实现边界不对
- 需要新增阶段或回退阶段
- 已完成内容需要大面积重做

不要把返工问题伪装成“小修小补”继续推进。

## 沟通原则

- 先给事实，再问反馈
- 先讲结论，再补验证证据
- 先讲限制，再谈是否进入 `finish`
- 不要省略已知风险

## 完成条件

`confirm` 阶段完成时，应满足以下之一：

- 用户明确确认通过，可以进入 `finish`
- 已识别出需要返工，并明确转入 `rework`
