### 三、正式审验的执行方式（待用户确认后执行）

#### 角色定位

- 主 Agent：担任总监 / 总工程师，负责拆分任务、编写子代理提示词、并行调度、审阅结果、汇总结论。
- 子代理：分别负责各自分段范围内的审验、证据收集、结论整理与分段报告撰写。

#### 工作原则

1. 主 Agent 不应亲自逐个文件做完整底层核查，而应将审验任务按模块合理拆分给多个子代理。
2. 子代理之间应尽量并行执行，以提升整体效率。
3. 分段应以**需求模块/主题边界**为主，而非简单机械按文件或行号平均切割。
4. 每个子代理都应围绕其负责模块给出：
   - 审验范围
   - 核查依据
   - 已实现内容
   - 未实现或实现不完整内容
   - 风险与影响
   - 证据引用

#### 子代理任务要求

每个子代理都应：

- 以 `task-optimized.50adab8.md` 中对应模块作为审验基准；
- 只进行调研、核查、分析与报告编写；
- 不直接修复问题；
- 不修改项目原有实现文件；
- 将各自分段结果保存为独立报告文件。

### 四、正式审验过程中的约束条件（待用户确认后执行）

1. **不得直接修复问题。**
   - 发现缺陷后只能记录，不得顺手修改实现。

2. **不得修改项目内原有实现文件。**
   - 包括源码、配置、已有说明文档、既有任务文档等。
   - 允许新增或写入本次审验专用报告文件。

3. **允许新增的文件仅限审验报告类产物。**
   - 分段报告：`reports/rpXXX.md`
   - 汇总报告：`report-summary.md`

4. **报告内容必须有依据。**
   - 结论应尽量引用对应文件、关键逻辑、配置、命令定义或文档位置作为证据。

5. **审验结论应区分状态。**
   - 可按“已实现 / 部分实现 / 未实现 / 无法确认”分类。

6. **审验必须覆盖完整基准。**
   - 不因用户已指出某一已知问题，就省略其他模块的核查。

## 分析发现

### 已识别的关键要求

1. 本次任务不是立即审验，而是**先打磨任务描述**。
2. 审验基准已经明确：`task-optimized.50adab8.md`。
3. 用户已知晓一个问题，但要求进行**完整范围审验**。
4. 审验执行方式必须体现“总工程师调度多个子代理并行完成”的协作模式。
5. 审验阶段只能产出报告，不能修改实现。

### 需要重点防止的偏差

1. **跳过用户确认直接开工**
   - 这是当前最需要避免的流程错误。

2. **只检查已知问题，不检查全量需求**
   - 会导致审验范围缩小，违背用户要求。

3. **主 Agent 亲自细查全部文件，未真正使用子代理并行分工**
   - 不符合用户指定的角色定位。

4. **发现问题后顺手修复实现**
   - 明确违背用户约束。

5. **分段报告写得过于空泛、缺少证据**
   - 会降低最终汇总结论的可信度。

### 风险与注意事项

1. `task-optimized.50adab8.md` 规模较大，需求覆盖面广，正式审验时需要合理拆分，否则单个子代理任务会过重。
2. 某些要求可能表现为“文档存在但程序未实现”或“程序部分实现但未满足完整规范”，需要在报告中区分“部分实现”与“未实现”。
3. 某些约束是行为性要求（如 Agent 不应直接读取某些文件、应通过 aide 程序获取信息），正式审验时需结合代码、文档、命令设计综合判断，不能只看单一文件。
4. 已知缺陷“5.3 场景预设（Presets）”可能只是更大范围遗漏的一部分，正式审验时应检查其上下游关联实现是否也存在缺口。

## 复杂度评估

| 维度 | 评估 | 说明 |
|------|------|------|
| 结构复杂度 | 高 | 审验基准涵盖目录结构、命令、skills、程序逻辑、工作流与文档多类对象 |
| 逻辑复杂度 | 高 | 需要判断“是否实现”“实现是否完整”“是否符合已确认决策” |
| 协作复杂度 | 高 | 用户要求必须通过多个子代理进行合理分段、并行审验与汇总 |
| 风险等级 | 中 | 主要风险来自审验遗漏、分工不当、证据不足，而非代码变更风险 |

**建议处理方式**：

- 当前先停留在任务打磨阶段；
- 待用户确认 `task-optimized.md` 后，再进入正式审验；
- 正式审验时优先按模块拆分为多个并行子任务；
- 每个子任务先产出独立证据化报告，最后再统一汇总。
