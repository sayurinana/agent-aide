## 6. 发现的问题

### 问题 1：`bye` 未形成基准要求中的结束任务闭环

基准要求 `bye` 在“任务分支且任务已到结束阶段”时，需要完成任务结束动作并合并回常驻分支；当前实现只做到：必要时 `verify/confirm/archive`，然后执行 `aide bye` 切回常驻分支。`aide archive` 也只做归档，不做合并。

**影响**：`bye` 目前更像“会话级离场”，不是基准要求中的“任务结束收尾入口”。

### 问题 2：`make-memory` / `load-memory` 的实际 memory 产物未落地

虽然 command 与 skill 已迁移到新体系，但当前仓库没有完整的 `aide-memory/memory/` 文档集，仅见占位 `overview.md`。

**影响**：

- `make-memory` 的成果标准没有在当前仓库落地
- `load-memory` 无法按其自身要求完成标准载入
- `hi/go` 中依赖 memory 的建议与接续能力会受限

### 问题 3：命令入口命名存在不统一

当前新文档大量使用 `/aide:make-memory`、`/aide:hi`、`/aide:go`、`/aide:bye` 这类插件式命名；但当前项目实际同步到 Claude Code 的文件位于 `/repo/agent-aide/.claude/commands/*.md` 顶层，并非 `aide/` 子目录。

按当前仓库内的命令组织方式，项目级入口更接近 `/make-memory`、`/load-memory`、`/hi`、`/go`、`/bye`，而不是统一的 `/aide:*`。

**影响**：用户容易按文档输入错误命令名，或对“插件命令”和“项目级同步命令”的入口产生混淆。

### 问题 4：旧体系残留文档仍较多，且与新体系并存

仓库中仍保留大量旧体系内容，例如：

- `/repo/agent-aide/docs/reference/aide-overview.md`
- `/repo/agent-aide/docs/reference/project-details.md`
- `/repo/agent-aide/aide-plugin/docs/commands/docs.md`
- `/repo/agent-aide/aide-plugin/docs/commands/load.md`

这些内容仍在描述 `/aide:setup`、`/aide:load`、`/aide:docs`、`/aide:run`、`.aide/project-docs/` 等旧体系对象。`/repo/agent-aide/aide-plugin/docs/README.md` 也明确说明“旧版 command 文档目前仍保留作迁移参考”。

**影响**：如果审验时不主动排除，极易把旧体系残留误判成新体系已实现。

## 8. 风险与影响

### 8.1 对用户操作的风险

由于新文档大量写作 `/aide:*`，而当前项目级 `.claude/commands/` 的实际落地方式是顶层文件，用户在 Claude Code 中容易：

- 输入错误命令名
- 混淆“项目同步命令”和“插件命令”两种入口
- 误以为当前仓库已经提供统一的 `/aide:*` 入口

### 8.2 对任务接续与项目认知的风险

当前仓库缺少完整的 `aide-memory/memory/` 文档集，会直接削弱：

- `load-memory` 的可靠性
- `hi` 结合 memory 进行建议判断的准确性
- `go` 接续任务时恢复上下文的完整性

### 8.3 对任务收尾质量的风险

`bye` 未形成“任务结束 → 完成收尾 → 合并回常驻分支”的闭环，意味着：

- 完成态任务的最终收尾仍需人工补流程
- 任务分支可能归档后仍未并回常驻分支
- 实际工作流与基准定义不一致，容易造成用户预期偏差

### 8.4 对后续维护与审验的风险

旧体系文档与新体系文档并存，会导致：

- 新成员误读旧文档
- 审验时误把旧体系当成新体系实现
- 文档维护成本与一致性风险上升

## 审验结论（简述）

当前仓库在本次范围内已经完成了**新体系 command/skill 的文档迁移与定义落地**，`hi` 与 `go` 还具备较完整的 `aide` 程序实现与测试支撑；但 `bye` 与基准要求仍有关键差距，`make-memory/load-memory` 也尚未在当前仓库形成完整可用的 memory 成果，因此整体应判为：**定义层基本到位，实际落地层仍为部分实现**。
