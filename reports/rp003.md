## 2. 审验依据

### 2.1 基准要求

本次审验的直接基准为：

- `/repo/agent-aide/task-optimized.50adab8.md:138-208`（3.2 子过程 Skills 定义）

### 2.2 判定口径

- **已实现**：仓库内存在对应 skill 文档，职责与新体系一致，且未见明显旧命名/旧流程残留影响其成立。
- **部分实现**：仓库内存在对应 skill 文档，但职责与程序支撑、相关文档或命名体系之间仍有断裂，或存在明显旧体系残留。
- **未实现**：仓库内未找到对应 skill，或仅有零散痕迹而不足以支持该 skill 成立。
- **无法确认**：仓库内证据不足，无法做出可靠判断。

## 3. 核查方法

1. 阅读基准文件，提取 3.2 中 9 个目标 skill 的职责、命名与阶段关系。
2. 在仓库内检索与 skill 相关的实际文件位置，重点检查：
   - `/repo/agent-aide/.claude/skills/`
   - `/repo/agent-aide/aide-plugin/skills/`
   - `/repo/agent-aide/.agents/skills/`
3. 对每个目标 skill 阅读 `SKILL.md`，核对其职责是否符合基准要求。
4. 继续核查与这些 skill 强相关的程序与全局文档，包括：
   - `/repo/agent-aide/aide-memory/aide-process-overview.md`
   - `/repo/agent-aide/.claude/commands/*.md`

   - `/repo/agent-aide/aide-memory/config.*`
5. 额外检索旧命名与旧阶段体系残留，例如：
   - `task-parser`
   - `task-optimize / flow-design / impl / verify / docs`
   - `aide flow next-part / back-part / issue / error / back-confirm`
6. 基于以上证据，对每个 skill 给出“已实现 / 部分实现 / 未实现 / 无法确认”结论。

## 4. 结论总览

### 4.1 总体判断

当前仓库内，3.2 的 9 个子过程 skill **均已具备明确的 skill 文档实体**，新阶段命名也已经在多处文档与部分程序中落地；但整体状态仍不是“完全实现”。

更准确的结论是：**新体系的子过程 skills 文档骨架已建立，但 build-task / make-graphics / finish / rework 仍存在明显的程序支撑缺口或旧体系残留，尚不能认定为全部完成。**

### 4.2 分类汇总

| 分类 | 项目 |
|------|------|
| 已实现 | `impl-verify`、`integration`、`review`、`docs-update`、`confirm` |
| 部分实现 | `build-task`、`make-graphics`、`finish`、`rework` |
| 未实现 | 无 |
| 无法确认 | 无 |

## 8. 风险与影响

1. **对使用者的直接误导风险高。**
   当前仓库同时存在：
   - 新 skill 文档
   - 新 `aide flow` CLI
   - 旧阶段说明文档
   - 旧命令示例与旧命名模板
   使用者或子代理很容易读到互相冲突的说明，导致按错流程执行。

2. **build-task → make-graphics → verify 链路存在断裂风险。**
   如果按 `make-graphics` 当前文档理解 `design.md` 标记，而不是按 `build-task` / `aide verify` 的 HTML 注释格式执行，后续审验会失配。

3. **finish 阶段无法稳定承担“正式收尾归档”职责。**
   当前 `aide flow` 主链路没有完成分支收束、归档、memory 同步的全流程能力，意味着 finish 仍需依赖人工理解与补动作，无法视为真正闭环。

4. **rework 阶段难以满足可追溯要求。**
   当前 `aide flow back` 不接收返工原因，也没有稳定落盘返工原因/影响分析，后续接续任务时可能只看见“回退了”，看不见“为什么回退”。

5. **阶段预设偏差会把 build-task 的阶段选择职责引向错误流程。**
   这会进一步影响是否进入 `make-graphics`、`review`、`docs-update`，也会影响 finish 前的质量控制路径。

6. **旧 `task-parser` 残留会削弱 build-task 重命名的可信度。**
   在总工程师/子代理协作模式下，这类旧入口残留极易让执行体走回旧体系，造成阶段职责混用。
