## 5. 逐项检查结果

### 5.1 情境（Situations）

**结论：部分实现**

#### 已实现部分

1. `aide hi` 能区分当前分支属于：
   - 常驻分支
   - 任务分支
   - 其他分支
2. 在常驻分支下，`aide hi` 会根据 Git 是否干净、是否存在未归档任务给出不同展示。
3. 项目中确实存在“草案目录”概念：`aide-memory/tasks/task-now/`，且 `aide verify`/`aide confirm` 会围绕该目录工作。

#### 未完全实现部分

1. **未形成基准要求的 4 类情境模型**。
   - 代码中没有显式的 `Situation` 类型或等价状态对象。
   - `aide hi` 只是按“当前分支类型 + 是否有任务 + Git 是否干净”临时输出结果，并非基于“可叠加情境”建模。

2. **“草案进行中”未被完整纳入状态判断**。
   - 基准要求：`task-now.md` 有变更，或 `task-now/` 目录存在，即属于“草案进行中”。
   - 当前 `aide hi` 没有检查 `task-now.md` 是否已修改，也没有检查 `aide-memory/tasks/task-now/` 是否存在。
   - 因而在“无未归档任务 + Git 干净”的情况下，仍可能输出“当前状态干净”，与基准不一致。

3. **“干净状态”的判断条件不完整**。
   - 基准要求同时包含：配置有效、git 可用且干净、位于常驻/任务分支、无草案残留。
   - 当前实现只覆盖了其中的部分条件，且主要体现在 `aide hi` 的展示逻辑中。

4. **“任务实施中”只有间接体现，没有明确情境输出**。
   - 当前通过“未归档任务列表”间接表示有任务在进行，但没有显式输出“任务实施中”这一情境名称。

#### 判断

因此，5.1 不是完全未做，而是实现了“部分状态诊断能力”；但距离基准要求的完整情境体系还有明显差距。

### 5.2 阶段（Phases）

**结论：部分实现**

#### 已实现部分

1. **新阶段集合已在程序中落地**。
   - `FlowPhase` 已定义 `build-task`、`make-graphics`、`impl-verify`、`integration`、`review`、`docs-update`、`confirm`、`finish` 八个阶段。

2. **`todo.md` 中的阶段流程声明已被程序提取和管理**。
   - 程序会从 `todo.md` 的 `<!-- PHASES: ... -->` 注释中提取阶段序列。
   - 支持 `impl-verify:loop` 标记。
   - 要求阶段中必须包含 `confirm`，并且必须以 `finish` 结束。

3. **`aide verify` 已对阶段声明做基础校验**。
   - `todo.md` 必须以 `build-task` 开始。
   - 必须存在复选框任务点。

4. **`aide flow` 已实现阶段查看、推进、回退的基本支持**。
   - 当前公开 CLI 为 `flow status / next / back / list / show`。
   - `flow next` 会进入下一阶段。
   - `flow back <phase>` 会回退到更早阶段，并提示后续需要重新经过的阶段。

5. **测试已覆盖新阶段流转的核心路径**。
   - 已有测试验证：从 `todo.md` 初始化阶段、向下一阶段推进、回退到更早阶段、`preset` 持久化。

6. **文档层面对新阶段体系已有支撑**。
   - `aide-memory/aide-process-overview.md` 已按新阶段体系描述典型流程。
   - `build-task` skill 已要求在 `todo.md` 中声明阶段流程，并说明可按任务特性插入可选阶段。

#### 存在的问题与偏差

1. **默认配置仍保留旧阶段体系**。
   - `aide-memory/config.toml` 与 `aide/src/core/config.rs` 默认 `flow.phases` 仍为：
     `task-optimize -> flow-design -> impl -> verify -> docs -> confirm -> finish`
   - 这与基准要求的新阶段体系不一致。

2. **对外文档仍有明显旧体系残留**。
   - `aide/docs/README.md` 仍展示 `aide flow start / next-step / next-part` 示例。
   - `aide/docs/commands.md` 仍系统性描述旧的 `start / next-step / back-step / next-part / back-part / back-confirm / issue / error / clean` 体系。
   - 但 `main.rs` 当前公开的 `aide flow` 子命令只剩 `status / next / back / list / show`。
   - 说明程序接口与文档已发生分叉。

3. **返工规则与基准存在偏差**。
   - 基准 5.2 写的是“从任意阶段可返回 `build-task` 发起返工”。
   - 当前 `flow back` 实现为：**可回退到任意已启用的更早阶段**，不局限于 `build-task`。
   - 这不是“完全缺失”，但与基准文本不完全一致。

4. **阶段顺序约束未被程序完全硬编码**。
   - 程序对 `confirm` 存在、`finish` 结尾有硬校验；`build-task` 起始约束主要在 `aide verify` 中实现。
   - 但对于可选阶段的相对插入位置，程序并未全面强制，只是通过 preset 和文档建议体现。

#### 判断

5.2 已经有较明显的落地实现，但存在“新阶段能力已上线、默认配置和部分对外文档仍停留在旧体系”的并存问题，因此应判定为**部分实现**而非“已实现”。

### 5.3 场景预设（Presets）

**结论：部分实现**

#### 已实现部分

1. 程序内已定义 `FlowPreset`：
   - `full`
   - `standard`
   - `lite`
   - `docs`
   - `research`
   - `custom`

2. 已实现 **preset 自动识别**。
   - 程序会根据 `todo.md` 中解析出的阶段序列自动识别当前任务属于哪个 preset。
   - 识别结果会写入 `flow-status.json`。

3. 测试已覆盖 preset 识别逻辑。

#### 核查结果：仅 `full` 与基准一致，其余 4 个预设均不一致

| 预设 | 基准要求 | 当前实现 | 结论 |
|------|----------|----------|------|
| `full` | `build-task -> make-graphics -> impl-verify:loop -> integration -> review -> docs-update -> confirm -> finish` | 与基准一致 | 已实现 |
| `standard` | `build-task -> impl-verify:loop -> review -> confirm -> finish` | `build-task -> impl-verify:loop -> integration -> review -> docs-update -> confirm -> finish` | 不一致 |
| `lite` | `build-task -> impl-verify -> confirm -> finish` | `build-task -> impl-verify:loop -> confirm -> finish` | 不一致 |
| `docs` | `build-task -> impl-verify -> review -> confirm -> finish` | `build-task -> impl-verify:loop -> docs-update -> confirm -> finish` | 不一致 |
| `research` | `build-task -> make-graphics -> impl-verify -> docs-update -> confirm -> finish` | `build-task -> impl-verify:loop -> review -> confirm -> finish` | 不一致 |

#### 其他问题

1. **preset 只有“检测”，缺少明确“选择入口”**。
   - 当前实现主要是根据阶段序列“反推 preset”。
   - 但没有发现明确的 CLI 或任务文档字段用于显式选择 `full/standard/lite/docs/research`。

2. **preset 缺少用户可见展示**。
   - `flow-status.json` 会保存 `preset` 字段，但 `aide flow status` 并不会把 preset 名称展示给用户。

3. **文档支持不足**。
   - `aide-memory/aide-process-overview.md` 说明了典型阶段流，但没有把 5 个 preset 作为一组正式预设向用户展示。
   - `build-task` skill 说明了如何按任务特性插入阶段，但未把 `full/standard/lite/docs/research` 作为标准化入口写清。

#### 判断

因此，5.3 不是“未实现”；程序内部已经有 preset 枚举、检测与持久化机制。但由于 **定义错误、展示不足、入口不完整**，整体只能判定为**部分实现**。

### 5.4 Agent 角色定位

**结论：部分实现**

#### 已实现部分

1. **角色定位在文档层面已较完整落地**。
   - `aide-memory/AGENT.md` 已明确将 Agent 定位为“统筹全局的总工程师”，并说明：
     - 先理解全局
     - 规划路径
     - 分配执行
     - 最终验收与整合
   - 文档还明确了“专家执行单元”的边界与适用场景。

2. **已考虑“宿主不支持子代理”的退化策略**。
   - `AGENT.md` 明确写出：若宿主环境不支持真正的子代理，则按“总工程师 + 专家边界”的方式模拟分工。
   - 这与当前工具环境实际情况是相容的。

3. **command 文档已围绕总工程师模型编写**。
   - 例如 `make-memory` command 明确要求：若宿主支持子代理，则先创建只负责 memory 的专家代理；否则由总工程师按边界自行执行。
   - `hi/go/bye` command 也都要求先理解总览文档，再根据状态选择命令与 skill。

4. **初始化流程会创建/同步角色文档**。
   - `aide init` 会创建或从全局仓库复制 `aide-memory/aide-process-overview.md` 与 `aide-memory/AGENT.md`，说明该角色体系已被纳入工程初始化产物。

#### 未完全实现部分

1. **程序层面没有子代理调度能力**。
   - 当前 `aide` CLI 提供的是项目初始化、任务确认、归档、状态查看、阶段推进等能力。
   - 未发现用于“创建专家子代理 / 指派子代理 / 跟踪子代理结果”的程序接口。

2. **“用户 / 总工程师 Agent / 专家子代理”三方职责分工主要停留在文档和提示词层**。
   - 即：角色模型已形成，但主要依赖 Agent 遵循文档执行，而不是由程序强约束。

#### 判断

5.4 的“定位”本身并非完全未做，相反其文档体系已经比较完整；但它还没有变成程序可执行的协作调度能力，因此应判定为**部分实现**。

## 6. 发现的问题

1. **新旧两套工作流体系并存，存在明显不一致**
   - 新体系：`build-task / make-graphics / impl-verify / integration / review / docs-update / confirm / finish`
   - 旧体系：`task-optimize / flow-design / impl / verify / docs / confirm / finish`
   - 当前程序、默认配置、测试、对外文档没有完全同步。

2. **`aide hi` 的情境判断不足以覆盖基准要求**
   - 不能完整判断“草案进行中”。
   - 也没有把四类情境建模为可叠加的正式状态。

3. **preset 定义错误较多，问题已经被基准明确指出且本次复核确认属实**
   - 仅 `full` 与基准一致。
   - `standard`、`lite`、`docs`、`research` 均与基准不符。

4. **preset 机制缺少对外入口和展示**
   - 当前更像“内部识别标签”，不是完整的面向用户/Agent 的工作流入口机制。

5. **返工规则与基准文本存在偏差**
   - 基准以“回到 `build-task` 发起返工”为中心。
   - 当前实现允许回退到任意更早已启用阶段。

6. **Agent 角色定位主要停留在文档层，程序不具备子代理调度能力**
   - 这意味着“总工程师 + 专家子代理”更多是操作约定，不是程序能力。

7. **旧的 FlowTracker/Hook 代码仍保留旧阶段名痕迹**
   - 例如 `flow-design`、`docs` 等旧命名仍出现在旧流程实现中，进一步说明体系切换尚未彻底收口。

## 8. 风险与影响

1. **工作流理解分裂风险**
   - 用户、Agent、文档、默认配置、程序实现所依据的阶段体系不完全一致。
   - 影响：容易导致 task 文档按新体系编写，但用户查看旧文档或配置时获得旧认知，产生误用。

2. **preset 误判风险**
   - 由于多数 preset 定义与基准不符，程序自动识别结果可能与设计预期不一致。
   - 影响：阶段流转、质量门槛、是否进入 review/docs-update 等环节可能被错误引导。

3. **情境判断失真风险**
   - `aide hi` 未覆盖“草案进行中”等关键状态。
   - 影响：可能在仍有 draft 残留时被误判为“干净”，从而影响后续 `confirm/go/bye` 决策。

4. **返工路径不一致风险**
   - 基准强调回到 `build-task` 发起返工，而当前实现允许回到任意更早阶段。
   - 影响：团队成员可能对“返工是否必须重新做任务构建”产生不同理解。

5. **角色体系停留在操作约定层的风险**
   - 当前“总工程师 + 专家子代理”主要依赖文档约束，而非程序能力。
   - 影响：执行质量高度依赖具体 Agent 是否严格遵守提示词，难以通过程序保证一致性。

## 最终结论

当前项目对“工作流体系设计”已经有实质性落地：新阶段枚举、阶段流转、`todo.md` 提取、角色文档、部分 skill/command 文档均已存在；但整体仍处于**部分实现**状态，核心原因是：

- 情境体系没有完整建模；
- preset 机制虽然存在，但定义错误较多；
- 新旧工作流体系在配置、文档、旧代码中并存；
- Agent 角色定位主要落在文档层，程序未真正具备子代理协作调度能力。
