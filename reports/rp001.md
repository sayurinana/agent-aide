## 5. 逐项检查结果

### 5.1 数据目录迁移到 `aide-memory/`

**结论：部分实现**

已实现部分：

- `ConfigManager` 以 `aide-memory` 为项目数据根目录，并在初始化时创建 `tasks`、`archived-tasks`、`templates`、`memory/structure`、`memory/concepts`、`memory/diagram` 等目录。
- 当前仓库实际存在 `/repo/agent-aide/aide-memory/`，且包含 `config.toml`、`config.md`、`branches.json`、`branches.md`、`tasks/`、`archived-tasks/`、`templates/`、`memory/` 等结构。

未完全实现部分：

- 仓库内仍保留多处指向旧 `.aide/` 数据结构的文档，说明迁移没有彻底完成，容易造成使用与理解偏差。

判定理由：目录结构本身已落地，但文档体系没有完全同步，不能判定为完全实现。

### 5.2 `task-now.md` 位于项目根目录

**结论：已实现**

核查结果：

- 当前仓库根目录存在 `/repo/agent-aide/task-now.md`。
- 初始化逻辑会按照 `task.description_file` 配置在项目根目录创建任务描述文件，默认值为 `task-now.md`。
- 任务确认后，程序会将任务描述文件重置回模板内容。
- 集成测试明确验证了 `task-now.md` 的创建与确认后重置行为。

### 5.3 `task-now/` 位于 `aide-memory/tasks/`

**结论：已实现**

核查结果：

- `TaskCommandContext::draft_dir()` 明确将草案目录定位为 `aide-memory/tasks/task-now`。
- `aide verify` 会直接审验该目录；`aide confirm` 会将其重命名为 `task-{n}`。
- 集成测试验证了 `aide confirm` 后 `aide-memory/tasks/task-1` 存在且 `aide-memory/tasks/task-now` 已消失。

### 5.4 `config.toml` 的访问约束：Agent 不可直接读取，必须通过 aide 程序获取

**结论：部分实现**

已实现部分：

- 项目已提供程序化配置访问通道：`aide config get <key>`，配置说明文档也引导使用该命令读取配置。
- `.claude/skills/aide/SKILL.md` 中明确写有“配置文件 Agent 不可直接读取 config.toml，可通过 config.md 了解配置项”。

未实现部分：

- 程序内部大量直接读取 `aide-memory/config.toml`，说明“不可直接读取”并未形成统一的技术隔离层，只是对 Agent 的行为约束描述。
- 仓库中未见任何面向 Agent 的硬约束机制，例如封装后的只读接口、访问代理、专门防误读策略或文档层面的统一禁读声明系统。

判定理由：存在替代通道和文档约束，但缺乏真正的访问隔离，只能认定为部分实现。

### 5.5 `config.md` 的访问约束：Agent 可在任务涉及配置理解时读取

**结论：已实现**

核查结果：

- `/repo/agent-aide/aide-memory/config.md` 已生成并完整说明配置项。
- 文档中明确给出 `aide config get` / `set` / `reset` / `update` 的使用方式。
- `.claude/skills/aide/SKILL.md` 也允许通过 `config.md` 了解配置项。

### 5.6 `branches.json` 的访问约束：Agent 不应直接读取，应通过 aide 命令查询

**结论：部分实现**

已实现部分：

- 程序中 `aide hi` 的任务收集逻辑会读取 `aide-memory/branches.json`，作为对外查询分支信息的正式通道。
- `branches.json` 与 `branches.md` 由 `save_branches_data()` 统一保存，`aide confirm` / `aide archive` 会自动更新它们。

未实现部分：

- 程序虽然提供了通过 `aide` 命令查询的路径，但并未对直接读取 `branches.json` 形成访问限制。
- `.claude/skills/aide/SKILL.md` 中仍直接表述“从 `branches.json` 查询对应分支名”，与“Agent 不应直接读取，应通过 aide 命令查询”的基准要求不完全一致。

判定理由：程序能力已经具备，但约束未闭环，且技能文档存在偏差，因此只能判定为部分实现。

### 5.7 `branches.md` 的访问约束：Agent 不可读取

**结论：未实现**

核查结果：

- `/repo/agent-aide/aide-memory/branches.md` 已存在，且由程序自动生成与更新。
- 但仓库中未见任何程序逻辑或统一规范机制阻止 Agent 读取该文件。
- 也未见专门的访问保护、替代接口提示、或强制通过 aide 命令获取同类信息的统一限制实现。

判定理由：只有文件存在和自动维护，不等于“不可读取”要求已落地。该约束目前没有程序化或流程化实现，应判定为未实现。

### 5.8 图解标记：不使用 `.no_graphics` 文件

**结论：已实现**

核查结果：

- 全仓库检索未发现当前实现中有 `.no_graphics` 的实际使用。
- 当前 `aide verify` 完全依赖 `design.md` 中的图解标记来判断是否需要图解。
- 基准文件中虽在 `aide verify` 描述段曾出现“`.puml 文件（或 .no_graphics 标记）`”的旧表述，但当前代码实现已经转向 `design.md` 标记，不再依赖 `.no_graphics`。

### 5.9 图解标记：在 `design.md` 中使用特定格式表示无需图解，并记录原因

**结论：已实现**

核查结果：

- `aide verify` 会调用 `parse_graphics_directive()`，识别 `<!-- GRAPHICS: required -->` 或 `<!-- GRAPHICS: skip: 原因 -->`。
- 当 `design.md` 缺少图解标记时，程序会报错 `design.md 缺少图解标记`。
- 当标记为 `skip` 且带有原因时，程序会记录“design.md 已标记无需图解：{reason}”。
- `.claude/skills/build-task/SKILL.md` 已把该标记格式写成明确规范。
- 集成测试已覆盖“缺少图解标记时报错”“带 skip 原因时验证通过”等场景。

## 6. 发现的问题

### 问题 1：访问约束主要停留在说明层，未形成完整的实现闭环

- `config.toml`、`branches.json`、`branches.md` 的访问约束中，只有“通过 aide 命令/通过 config.md 获取”的替代路径较明确。
- 但对“不允许直接读取”“不应直接读取”“不可读取”的限制，没有对应的程序化隔离或统一机制。
- 其中 `branches.md` 的“Agent 不可读取”要求最弱，基本没有落地证据。

### 问题 2：技能文档与基准约束存在偏差

- `.claude/skills/aide/SKILL.md` 第 32 行仍写“从 `branches.json` 查询对应分支名”，容易把 Agent 引导为直接面向数据文件，而非通过 aide 命令查询。
- 这与基准文件第 69 行“`branches.json`：Agent 不应直接读取，应通过 aide 命令查询分支信息”不完全一致。

### 问题 3：目录结构迁移虽已落地，但文档迁移不彻底

- 仓库中仍存在多份旧 `.aide/` 结构说明，例如：
  - `/repo/agent-aide/aide/docs/data-formats.md`
  - `/repo/agent-aide/docs/reference/aide-overview.md`
- 这些文档继续描述 `.aide/config.toml`、`.aide/branches.json`、`.aide/flow-status.json` 等旧路径，会削弱“数据目录结构已调整为 `aide-memory/`”这一变更的确定性。

## 8. 风险与影响

1. **访问边界不清风险**
   - 若 Agent 是否允许读取 `config.toml`、`branches.json`、`branches.md` 仅靠口头约束，而非统一机制控制，后续 command/skill/文档编写容易继续偏离基准要求。
   - 直接影响审计一致性、行为一致性和后续子代理协作边界。

2. **文档与实现不一致风险**
   - 旧 `.aide/` 文档残留会让用户、维护者或子代理误判当前真实数据结构。
   - 在正式流程中，这会导致错误引用路径、错误理解配置键和错误操作数据文件。

3. **`branches.md` 约束失效风险**
   - 基准明确要求该文件“Agent 不可读取”，但当前没有落地控制，意味着该要求在实际执行中基本不可依赖。
   - 如果后续流程把该要求视为前提，可能造成审验结论、执行规范与实际行为脱节。

4. **结构迁移完成度被高估的风险**
   - 从目录和代码看，新结构已基本成形；但从规范一致性看，仍有明显残留与偏差。
   - 若宽松判定为“已实现”，会掩盖后续规范治理工作量，影响后续审验准确性。

## 最终判定

针对本次要求重点审验的“数据目录结构调整”范围，结论为：**部分实现**。

核心原因如下：

- 目录位置、任务草案路径、图解标记机制、配置与分支文件生成维护等，已有较强实现基础。
- 但访问约束，尤其是 `branches.md` 不可读取、`branches.json` 应通过 aide 命令查询等要求，没有形成完整落地。
- 同时，仓库文档中仍残留多处旧 `.aide/` 结构说明，说明数据目录迁移尚未完全收敛。
