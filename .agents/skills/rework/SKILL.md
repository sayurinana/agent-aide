# rework 指南

## 目标

当任务执行中发现问题需要回退时，规范地处理返工流程，确保：
- 返工原因和影响被充分记录
- 返工后至少经过必要的后续阶段
- 上下文不丢失，可追溯返工历史

## 返工触发条件

出现以下情况时应考虑返工：

| 触发条件 | 说明 |
|----------|------|
| 需求理解偏差 | information.md 的目标与用户实际需求不符 |
| 设计方向错误 | design.md 的方案不可行或严重偏离 |
| 实现边界问题 | 完成后发现遗漏关键场景 |
| 需要新增阶段 | 原阶段流程不够，需要补充 |
| 需要删除阶段 | 原阶段流程冗余，需要简化 |
| 大面积重做 | 已完成内容需要推翻重来 |

不要把返工级问题伪装成"小修小补"。

## 阶段体系

新体系下的阶段名称：

| 阶段 | 类型 | 说明 |
|------|------|------|
| build-task | 固定 | 任务构建与需求完善 |
| make-graphics | 可选 | 图解绘制 |
| impl-verify | 固定 | 实施与验证循环 |
| integration | 可选 | 集成测试 |
| review | 可选 | 代码审查/文档审校 |
| docs-update | 可选 | 文档更新 |
| confirm | 固定 | 用户确认 |
| finish | 固定 | 收尾归档 |

## 返工目标选择

根据问题类型选择返工目标阶段：

| 返工目标 | 触发条件 | 处理方式 |
|----------|----------|----------|
| build-task | 需求偏差、新需求 | 更新 information.md |
| make-graphics | 图解设计问题 | 更新 design.md |
| impl-verify | 实现细节问题 | 在当前阶段修复 |
| integration | 集成问题 | 更新并重新集成测试 |
| review | 审查发现问题 | 修复后重新审查 |

## 返工流程

### 1. 记录问题

在 todo.md 或创建返工记录文件：

```markdown
## 返工记录 [时间戳]

### 问题描述
[问题的具体描述]

### 发现阶段
[在哪个阶段发现]

### 影响分析
- 已完成项：[列表]
- 需重新处理：[列表]
- 新增需求：[如有]

### 返工目标
[目标阶段]

### 原因说明
[为什么需要返工到该阶段]
```

### 2. 更新相关文档

根据返工目标更新：

#### 返工到 build-task

- 更新 information.md（加入新需求或修正理解）
- 可能需要重新生成 design.md、todo.md

#### 返工到其他阶段

- 创建/更新返工需求文档
- 在 design.md 或 todo.md 中添加返工声明
- 梳理已完成、未完成、需重新处理项

### 3. 向用户确认

告知用户返工计划：

> 发现问题需要返工到 [目标阶段]。原因：[原因简述]。
> 影响范围：[影响分析摘要]。
> 返工后需要重新经过的阶段：[阶段列表]。
> 请确认是否同意返工。

### 4. 执行返工

用户确认后：

1. 更新 todo.md 标记返工状态
2. 清理或保留已完成内容（视情况）
3. 使用 `aide flow` 命令记录返工（如有）

### 5. 重启执行

返工确认后建议：

- 重启对话，避免上下文混乱
- 从返工目标阶段重新开始
- 按更新后的阶段流程执行

## 返工后必须经过的阶段

返工后必须至少经过后续必要阶段才能再次进入 confirm：

| 返工来源 | 必经阶段 |
|----------|----------|
| confirm → build-task | build-task → impl-verify → confirm |
| confirm → impl-verify | impl-verify → confirm |
| confirm → review | review → confirm |
| impl-verify → build-task | build-task → impl-verify |

可选阶段（make-graphics、integration、review、docs-update）视情况是否需要。

## 返工注意事项

### 上下文保留

返工前必须更新文档，确保：
- 问题原因被记录
- 已完成内容状态清晰
- 新需求或修正内容明确
- 不丢失任何关键信息

### 不要硬做

发现返工级问题时：
- 停止继续堆改动
- 不要试图在当前阶段"修补"
- 正式进入返工流程

### 诚实沟通

向用户如实说明：
- 问题是什么
- 影响有多大
- 返工需要做什么
- 预期时间和风险

## 与 aide flow 的配合

返工时可使用 aide flow 命令记录：

```bash
# 记录返工问题
aide flow issue "发现设计缺陷，准备返工"

# 阶段回退
aide flow back-part build-task "需求理解偏差，返工重新构建"
```

## 完成条件

rework 处理完成时，应满足：

- 返工原因已记录
- 相关文档已更新
- 用户已确认返工计划
- todo.md 已标记返工状态
- 明确返工后的阶段流程
